
**Какие команды Git ты используешь чаще всего?**

`git add <file> <file>` - Индексирование файлов
`git commit -m "<message>"` - Закомитить изменения с сообщением
`git push origin <branch>` - Выполнить push изменений на удаленный сервер
`git checkout <branch>` - Переключиться на другую ветку (`-b` - опция для создания ветки)
`git fetch` - скачать все возможные изменения с удаленного репозитория
`git pull origin <branch>` - сочетание команды `git fetch` и `git merge` (или `git rebase` в зависимости от настроек)
`git merge <branch>` - выполнить `merge` с указанной веткой 
`git rebase <branch>` - выполнить `rebase` с указанной веткой 
`git stash` - Припрятать локальные изменения без отправки в репозиторий


**Какие существуют политики слияния изменений?**

1) `merge commit`
Создает специальный коммит, который объединяет две истории
`[+]` - Можно отследить в какой момент какая фича была влита
`[-]` - Появляется "мусорный" коммит с сообщением о сливании

2) `fast-forward`
В отличии от `merge commit` - `fast-forward` не создает нового коммита, а просто перемещает указатель вперед. Возможен `ff` только при условии, что в целевой ветке не появилось дополнительных изменений. Иначе их необходимо подтянуть и снова поднять `MR`
`[+]` - Позволяет сделать четкую линейную структуру коммитов
`[-]` - Теряется информация о том, что какой-то коммит был сделан в другой фиче

3) `Squash and Merge`
Описание: Все коммиты из фичевой ветки объединяются в один коммит и встраиваются в основную ветку
`[+]` - Главная история последовательно состоит из фич, проще проводить код-ревью
`[-]` - Теряется информация по промежуточным этапам разработки фичи

4) `Rebase and Merge`
Описание: Выполняет слияние изменений с последней актуальной версией, накатывая изменения поверх основных
`[+]` Накатка идет всегда на последую и самую актуальную версию
`[-]` Происходит переписывание истории из фичевой ветки, так как новые создаются совершенно новые коммиты, если над одной фичой работает несколько человек, то нужно очень аккуратно производить слияние.
`[-]` При конфликтах придется решать конфликты для каждого отдельного коммта


**Что используется у тебя на работе: merge или rebase? Почему?**

У меня на работе используют `merge`, но я предпочитаю `rebase`.

Почему выбрали `merge` - в СБЕР строго относятся к `RN` - `Release Notes`. С merge и созданием отдельного коммита слияния чуть проще контролировать какая фича когда была влита и если расходятся `RN`, то проще понять проблему расхождения. Были мысли перейти на `Squash and Merge`, но решили не переходить, чтобы merge коммиты не были гиганскими.

Почему мне нравится `rebase`?
1) `rebase` не удобен при совместной работе над одной фичей, чтобы не было "общих" веток `rebase` заставляет все фичи дробить на более мелкие. Например, сели кто-то сделает `git push --force`, то все сломается у коллеги, которые работают в той же ветке
2) Четкая структура, код идет последовательно и ровно друг за другом, нет "мусорных" коммитов


**Что такое конфликт слияния? Расскажите, как ты решаешь конфликты слияния.**

Конфликт слияния - это ситуация когда над одним и тем же куском кода работало N разработчиков, что привело к конфликту слияния - git не понимает какую версию оставить.

Процесс:

Решения конфликта слияния зависит от того, какой функционал был задет, но обязательно с предупреждением человека, который вносил изменения в тот же кусок кода.

Если это что-то мелкое, по типу комментарий, документация, или логирования, то исправляю самостоятельно без обращения к коллегам, если же задевается крупный или важный функционал, то, во-первых, прошу коллегу подключиться к разбору конфликта, во-вторых, стараюсь продумать решение, которое не затронет и его фичу и мою 

Техника:

Все конфликты решаю при помощи `VSCode` - мой основной редактор, ориентируюсь маркерам `<<<<<<<`, `=======`, `>>>>>>>`, после того как конфликт решаен, выполняю стандартный `git add` и `git commit` для `merge` или `git rebase --continue` для `rebase`

**Как отредактировать коммит?**

Если ситуация требуется изменения последнего коммита в фиче ветке, то проще всего при помощи `git commit --amend` -  появится редактор, где можно будет внести изменения

Если речь про изменения файла коммита N коммитов назад, то процесс становится невероятно неприятным - так как требует переписывания истории, а переписывать историю умеет только `rebase`

1) Определяем номер коммита, который хотим изменить
```bash
>> git log -N # Посмотреть последние N изменений
```
2) Выполняем интерактивный `rebase`
```bash
>> git rebase -i HEAD~N
```
3) В появившемся оксе, меняем `pick` на `edit`
4) Внести изменения в файлы
5) Выполнить индиксацияю и изменение коммита
```bash
>> git add <file>
>> git commit --amend
```
3) Сохраняем и завершаем `reabse` 
```bash
>> git rebase --continue
```
7) Обязательно отправить в удаленный репозиторий с флагом `-f` или `--force`
```bash
>> git push origin --force
```
