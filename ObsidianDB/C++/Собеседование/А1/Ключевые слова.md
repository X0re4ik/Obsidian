
**Зачем нужно ключевое слово `static`?**

Статические объекты, это объекты, которые живут все время работы программы и в отличии от других переменных хранятся не в куче или на стеке, а в специальной области памяти `.bss` или `.data`.
Ключевое слово `static` применяется в 3 ситуациях:
1) Переменная или функция
В данном случаи `static` спасает нас от проблемы `ODR`, так как статические переменные считаются локальными для данной единицы трансляции. `static` создаёт внутреннее связывание (`internal linkage`), что делает переменные и функции невидимыми для внешних единиц трансляции.
2) Локальная переменная
Типичный пример это паттерн `singleton`, который позволяет создать только единственный объект при множественном вызове. Отмечу важный нюанс, стандарт гарантирует, что создание локальных переменных *потокобезопасно* (начиная с `C++11`)
3) Член класса
Здесь мы связываем переменную или метод неразрывно с именем класса, но сами переменные и функции не имеют доступа к данным экземпляра класса (`this` недоступен)

**Зачем нужно ключевое слово `inline`?**

Переменные:

1) Объявление статических переменных
`inline` в данном случаи позволил нам упростить процесс инициализации статических переменных
```C++
class A {
  static int a;
  inline static int b{0}; // Стало, начиная с C++17
};
int A::a{5}; // Было
```
2) Глобальные константы

До `C++17` при объявлении глобальной переменной необходимо было помнить, что для разных единиц трансляции будут созданы разные копии одной и той же переменной (3 определения  `const double PI=3.14159` - 3 копии в 3 `.cpp` файлах).
`inline` решил проблему и множественного определения и копирования. Линкер схлопывает несколько определений в один экземпляр, что позволяет единицам трансляции смотреть только на одну *общую* переменную.
Отметим, что в `C++17` переменные `constexpr` автоматически считаются `inline`

Функции:

Сделаем небольшой offtop. Ошибочно думать, что `inline` встраивает тело функции в код, так как по сути современные компиляторы и без inline решают, что вставить, а что нет.
А вот что точно inline умеют лучше всех - бороться с `ODR`. Функции как и глобальные переменные обладают *внешним связыванием*, а inline позволяет сказать линковщику: "Выбери любую реализацию этой функции"

Методы класса

Стоит отметить, что методы класса, которые объявлены и реализованы в `.hpp` файле и внутри тела класса автоматически получают `inline` статус.
```C++
// .hpp
class A {
public:
    int getOne() {return 1;}
    // inline int getOne() {return 1;}
};

// .hpp
class A {
public:
  int getOne();
};
// ODR. Необходимо inline int A::getOne() { return 1; }
int A::getOne() { return 1; }
```

Про *Undefined Behavior (UB)*

Ранее говорилось о том, что `inline` выбирает между несколькими реализациями. А что будет, если функция по какой то причине имеет разную реализацию в разных единицах трансляции. Например мы использовали одну и ту же библиотеку, но разных версий, что привело к двойственности.
Ответ грустный - какую реализацию выбрать ложиться на плечи линковщика - *Undefined Behavior*


**Зачем нужно ключевое слово `volatile`?**

`volatile` запрещает производить оптимизации с переменной. Подавляющее большинство ситуаций использования - изменение какой-либо переменной от внешнего мира, например, чтение датчиков, сигналы от других программ и т.п.
Важное уточнение - `volatile` никак не решает проблемы многопоточности, он не гарантирует атомарности и не выставляет барьеры памяти.

Проще всего роль `volatile` показать на примере:
```C++
// Без оптимизации
bool* sensor_flag = (bool*)0x4000FF00; // Аппаратный адрес датчика

void wait_for_signal() {
    while (*sensor_flag == 0) { 
        // Ждем, пока железо не выставит 1
    }
}
```
Мы читаем данные с датчика и, если данные, придут корректные, то выходим из цикла.
Но....
Компилятор может произвести оптимизацию `sensor_flag` по причине того, что переменная только читается (не изменяется)
```C++
// С оптимзацией
bool* sensor_flag = (bool*)0x4000FF00; // Аппаратный адрес датчика

void wait_for_signal() {
	if (*sensor_flag == 0) {
		while (true) {
			// Ждем, пока железо не выставит 1
		}
	}
}
```
Чтобы компилятор не оптимизировал любые взаимодействия с переменной - существует ключевое слово `volatile`
```C++
volatile bool* sensor_flag = (bool*)0x4000FF00; // Аппаратный адрес датчика
```


**Расскажи, как `auto`  выводит тип.**

Общее правило формирование типа `auto` идентично правилу вывода шаблонных параметров. Компилятор представляет `auto`, как шаблонный тип`T`

Общее правило формирование типа

1) `auto var = some_variable`
Если `some_variable` типа `T*` или `const T*` -> `auto` будет `T*` и `const T*`
Если `some_variable` типа `T`, `const T`, `T&`, `const T&` -> `auto` будет `T` 

2) `auto& var = some_variable`
Сильно зависит от типа `some_variable`, если это стандартная `lvalue` ссылка, то вывод типа будет `T&`, если же это `rvalue` - то ошибка компиляции. Но есть исключение - `some_variable` может быть `rvalue` при условии, что `var` имеет тип `const auto&` - `lifetime extension`

3) `auto&& var = some_variable`
Действует общее правило Скотта Мейерса про "универсальные ссылки": тип `var` будет зависеть от того какая `value category` у `some_variable`. Если `rvalue`, то тип `var` будет `T&&`, если же `lvalue`, то `T&`. `CV` - квалификаторы при этом сохраняются.

4) `auto var = {1, 2, 4}`
Исключение из общего правила - шаблон не сможет скомпилироваться, а `auto` успешно определит тип как `std::initializer_list<int>`.

Особенности работы с `lambda`

`auto` часто можно встретить с `lambda` выражением, во-первых, `lambda` функции невозможно определить кроме как `auto`, так как компилятор под каждую `lambda` функцию создает класс с оператором `operator()`. Так же лямбда функции можно встретить с аргументами `auto` - компилятор под капотом создает класс с уже шаблонным оператором вызова.

Пример:

```C++
class __AnonymousLambda {
public:
    template <typename T1, typename T2>
    auto operator()(T1 a, T2 b) const {
        return a + b;
    }
};

auto l = [](auto a, auto b){ return a + b;} // __AnonymousLambda()(a, b);
```


Про `decltype(auto)`

`decltype(auto)` позволяет сохранить семантику объекта справа. `auto` отбрасывает ссылки и cv -квалификаторы, `decltype(auto)` этого не делает
```C++
const int&& foo();
auto i = foo(); //  i будет иметь тип int
decltype(auto) i2 = foo(); //  i2 будет иметь тип const int&&
```
