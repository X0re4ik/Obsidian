










## Альтернативный способ подгрузки динамической библиотеки
Ранее мы рассмотрели процесс погрузки динамической библиотеки при старте приложения, но, если ее подгрузка нужна только тогда, когда реально используется тот или иной компонента?
Бональный, но актальный пример. В зависимости от входных параметров мы хотим выбрать тот или иной ML движок (кем бы он не был...)
## Пример (Динамическая подгрузка библиотеки)
Пусть имеется программы код формата:
```bash
.
└── mllib
    ├── mllib.cpp
    └── mllib.hpp
```
>`mllib.hpp` 
```C++
void sayILoveML();
```
>`mllib.cpp` 
```C++
#include <iostream>
void sayILoveML() { std::cout << "I Love ML" << std::endl; }
```
Задача:
1) Создать `main.cpp`, который сможет динамически подгрузить библиотеку `mllib`
2) Выполнить операцию `sayILoveML`

Решение:
1) Создаем динамическую библиотеку mllib:
```bash
>> g++ -fPIC -c ./mllib/mllib.cpp -o libmllib.o
>> g++ -shared libmllib.o -o libmllib.so
```
2) Создаем файл `main.cpp` формата:
```C++
#include <iostream>
#include <dlfcn.h> // Заголовочный файл для dlopen/dlsym/dlclose

int main() {
    std::cout << "Программа запущена. Загружаем библиотеку..." << std::endl;

    // 1. Открываем библиотеку (RTLD_LAZY означает ленивое связывание символов)
    void* handle = dlopen("./libplugin.so", RTLD_LAZY);
    
    if (!handle) {
        std::cerr << "Ошибка загрузки библиотеки: " << dlerror() << std::endl;
        return 1;
    }

    // 2. Объявляем указатель на функцию нужной нам сигнатуры
    // В нашем случае функция не принимает аргументов и возвращает void
    using PluginFuncType = void (*)();

    // 3. Ищем адрес функции по её имени
    // reinterpret_cast нужен, так как dlsym возвращает сырой указатель void*
    PluginFuncType my_func = reinterpret_cast<PluginFuncType>(dlsym(handle, "hello_from_plugin"));

    // Проверяем, не было ли ошибок при поиске символа
    const char* error = dlerror();
    if (error) {
        std::cerr << "Ошибка поиска символа: " << error << std::endl;
        dlclose(handle);
        return 1;
    }

    // 4. ВЫЗЫВАЕМ ФУНКЦИЮ!
    std::cout << "Функция найдена! Выполняем:" << std::endl;
    my_func();

    // 5. Закрываем библиотеку, освобождаем память
    dlclose(handle);
    std::cout << "Библиотека выгружена. Завершение программы." << std::endl;

    return 0;
}
```
3) Выполняем стандартную компеляцию и проверяем результат
```bash
>> g++ main.cpp -o main
>> ./main
Программа запущена. Загружаем библиотеку...
Ошибка поиска символа: ./libmllib.so: undefined symbol: sayILoveML
```
И снова мимо.... В чем же дело, почему мы не смогли вызвать функцию из нашей библиотеки?
Все дело в магическом `extern "C"`, которого у нас нет... C++ это в первую очередь про боль и преодаление... Наш любимый язык обладает такой замечательной вещью, как перегрузка функций:
```C++
// Что видит программист
int foo();
int foo(int);
int foo(float);
int foo(std::string);
```
Для программиста это 4 одинаковых функции с разной сигнатурой, для компилятора это 4 совершенно разных функции с разными названиями (компилятор каждой функции даст уникальный суффикс)
```C++
// Что видит компилятор
int foo1();
int foo2(int);
int foo3(float);
int foo4(std::string);
```
Механиз перегрузки C++ именуется `Name Mangling`, но в чистом `C` подобного механизма нет, а так как библиотека подгрузки динамических библиотек Си-шная, то и контракт необходимо соблюдать Си-шый.  `extern "C"` - говорит компилятору: Скомпилируй эту функцию по правилам языка `C`. Соотвественно, если мы обернем `mylib.cpp` в `extern "C"` , то подгрузка отработает успешно.
```C++
#include <iostream>
extern "C" {
	void sayILoveML() { std::cout << "I Love ML" << std::endl; }
}
```
Пересоберем библиотеку и запустим программу снова (Отметим, что перекомпиляция основной программы не нужна, так как линамические библиотеки всегда встраиваются в процессе работы)
```bash
>> g++ -fPIC -c ./mllib/mllib.cpp -o libmllib.o
>> g++ -shared libmllib.o -o libmllib.so
>> ./main
Программа запущена. Загружаем библиотеку...
Функция найдена! Выполняем:
I Love ML
Библиотека выгружена. Завершение программы.
```
