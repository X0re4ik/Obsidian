>Статическая библиотека — это архив предварительно скомпилированных объектных файлов

Статическая библиотека необходима в первую очередь для того, чтобы переиспользования компонентов.
Принцип простой, имя несколько объектных файлов, можно создать **архив** из них - это и будет наша статическая библиотека
Рассмотрим процесс создания статической библиотеки

### Пример 1 (Создание статической библиотеки)
Пусть есть 2 объектных файла: `./game.o` и `./utils.o`
```bash
ar r libmylib.a ./game.o ./utils.o
```
Утилита `ar` служит для создания статических библиотек, команда `r` указывает улитите, что необходимо выполнить `replace` `./utils.o` и  `./game.o` на `libmylib.a`.
По сути `libmylib.a` является архивом из 2-ух объектных файлов
Отметим, что есть **жесткое правило наименования** статических библиотек: `lib<НАЗВАНИЕ>.a`, то есть префикс `lib` и суффикс `.a` обязательны

### Пример 2 (Сборка простого проекта)
Пусть есть небольшой C++ проект со следующей файловой структурой (код библиотек приведён в комментариях для упрощения процесса чтения):
```bash
.
├── libmy1
│   ├── libmy1.cpp // int libmy1() {return 127;}
│   └── libmy1.hpp // int libmy1();
├── libmy2
│   ├── libmy2.cpp // int libmy2() {return 128;}
│   └── libmy2.hpp // int libmy2();
└── main.cpp
```
>`main.cpp`
```C++
#include "./libmy1/libmy1.hpp"
#include "./libmy2/libmy2.hpp"
#include <iostream>

int main() {
  int x = libmy1() + libmy2();
  std::cout << "Result: " << x << std::endl;
  return x;
}
```
Задача:
1) Создать объектные файлы libmy1.a и libmy2.a
2) Положить библиотеки в папку libs
3) Подключить их к объектному файлу main.o

Решение:
1) Создаём объектный файл libmy1.a и libmy2.a
```bash
>> g++ ./libmy1/libmy1.cpp -c -o libmy1.a
>> g++ ./libmy2/libmy2.cpp -c -o libmy2.a
```
2) Попытаемся выполнить компиляцию main.cpp без наших библиотек:

```bash
>> g++ main.cpp -o main.a
```
```bash
/usr/bin/ld: /tmp/cchcYXQm.o: в функции «main»:
main.cpp:(.text+0xe): неопределённая ссылка на «libmy1()»
/usr/bin/ld: main.cpp:(.text+0x15): неопределённая ссылка на «libmy2()»
collect2: error: ld returned 1 exit status
```
Очевидно, что мы упали на этапе линковки из-за ошибки `undefined reference`
3) Выполним компиляцию, указав в параметрах путь поиска наших статических библиотек и список наших библиотек
```bash
>> g++ main.cpp -o main.o -L./libs -lmy1 -lmy2
```
Вуаля... Программа успешно скомпилировалась. Разберем чуть поподробнее процесс:
* `L` - параметр поиска библиотек, именно по ней компилятор понимает откуда брать библиотеку
* `-l` параметр линковки библиотек, именно эти библиотеки будет линковать к проекту линковщик. Особое внимание стоит уделить тому, что название библиотек не соответствует тому, что мы указали в параметрах. Все из-за того, что компилятор знает о правиле нейминга статических библиотек и просит указать именно **название** библиотеки, а не ее полное имя
4) Выводим результат на экран
```bash
./main.o 
Result: 255
```
### Пример 3 (Работа в условиях двойного определения)
Вернемся к нашему проекту из примера 2. Пусть разработчик библиотеки `libmy2` назвал свою функцию `libmy1`.
```bash
.
├── libmy1
│   ├── libmy1.cpp // int libmy1() {return 127;}
│   └── libmy1.hpp // int libmy1();
├── libmy2
│   ├── libmy2.cpp // int libmy1() {return 128;}
│   └── libmy2.hpp // int libmy1();
└── main.cpp
```
Задача:
1) Разобрать ошибку компиляции
2) Предложить решение проблемы

Решение:
1) Повторим процесс создания библиотеки, опустив подробности операций
```bash
g++ ./libmy2/libmy2.cpp -c -o ./libs/libmy2.o
g++ ./libmy1/libmy1.cpp -c -o ./libs/libmy1.o
ar r ./libs/libmy.a ./libs/libmy1.o ./libs/libmy2.o
```
2) Попытаемся скомпилировать программу
```bash
g++ main.cpp -o main.a -L./libs/ -l my
```
Программа успешно **скомпилировалась**, хотя не должна...
Все дело в том, как компилятор получает необходимый объектный файл. Каждая статическая библиотека содержит таблицу символов, внутри которой содержится информация про вызову и функции. 
Посмотрим на содержимое нашей библиотеки:
> `nm ./libs/libmy.a`
```bash
libmy1.o:
0000000000000000 T _Z6libmy1v

libmy2.o:
0000000000000000 T _Z6libmy1v
```
Мы видим, что оба объектный файла содержат идентичные команды, но линковщик по своей природе ленив, он ищет первую подходящую реализацию необходимой ему функции. Таким образом он наткунлся на реализацию внутри `libmy1.o` и выполнил ее линковку (в `libmy2.o` он даже не заходил)
3) Давайте сломаем компиляцию, чтобы исключить подобные кейсы
```bash
g++ main.cpp -o main.a -L./libs/ -Wl,--whole-archive -lmy
```
Опция `-Wl,--whole-archive` загружает **ВСЕ** объектный файлы в библиотеки, в таком случаи компилятор явно скажет, что у нас есть 2 определения функции
```bash
... // Много непонятного кода
(.text+0x167): повторное определение «__morestack_set_guard»; /usr/lib/gcc/x86_64-linux-gnu/11/libgcc.a(morestack.o):(.text+0x167): здесь первое определение
/usr/bin/ld: /usr/lib/gcc/x86_64-linux-gnu/11/libgcc.a(morestack.o): в функции «__morestack_make_guard»:
(.text+0x171): повторное определение «__morestack_make_guard»; /usr/lib/gcc/x86_64-linux-gnu/11/libgcc.a(morestack.o):(.text+0x171): здесь первое определение
collect2: error: ld returned 1 exit status
```
