# Создание объектного файла
Предыдущие этапы дали нам самый важный файл: ассемблерный файл под нашу архитектуру.
На Linux за трансляцию ассемблерного текста (.s) в объектный файл (.o) отвечает утилита as (часто вызывается через gcc/clang как обёртку).
Рассмотрим чуть детальнее процесс создания объектного файла. Процесс создания объектного файла состоит из следующих этапов:
1) Лексический и синтаксический разбор asm кода
Стандартная операция проверки синтаксиса ассемблерного кода
2) Построение секций и размещение кода/данных
Ассемблер на данном этапе идет примерно по следующем алгоритму:
* Определяет директивы `.text`, `.data`, `.bss` и понимает, с какой секцией необходимо сейчас работать
* Весь код попадает в `.data`
* Все инициализации переменных в `.data`
* Простое резервирование под данные `.bss`
Все блоки попадают не в процессор напрямую, а в секции **объектного** файла
3) Формирование таблицы символов
Параллельно этапу *2* формируется таблица символов `.symtab`, данная таблица крайне важна, так как она говорит какой символ в какой секции находится и с каким смещением
4) Выполнение процесса релокации
Данный блок проще объяснить на примере. Пусть имеется ассемблерный файл:
```s
    .globl main
    .text

main:
    mov $msg, %rdi      # хотим положить адрес msg
    call puts           # хотим вызвать puts
    ret

    .data
msg:
    .asciz "Hello\n"
```
>`msg` определен
>`puts` не определен

Так как нет информации по `puts`, то совершенно неизвестен ее адрес, а также смещение, но ассемблеру все равно нужно сгенерировать объектный файл, так как функция `puts` может появится сильно позже (на этапе линковки)
Именно для таких функций создается файл запись с релокацией секций `.rela.text`

## Вывод
На данном этапе сгенерирован объектный файл, это набор секций с машинным кодом и он практически готов к выполнению, ему мешает лишь то, что часть инструкций еще не опредлено (таблица релокаций)

## Линковка
На данном этапе перед компилятором находится несколько N объектных файлов, каждый из которых имеет секцию данных, секцию кода и секцию релокации. Объектный файл сам по себе не *исполняемый* хотя бы по причине того, что часть ссылок на вызовы в нем не реализована (см. секцию релокации)
Процесс объединения нескольких объектных файлов именуется линковкой и выполняет ее линковщик
Рассмотрим поподробнее процесс линковки:
1) Линкер получает все объектный файл, в т.ч объектный файл библиотеки C++ (В ней храниться код наших `.cpp` STL)
2) Чтение секций и таблиц символов
3) Построение карты "Кто вызывает" -> "Кто определяет"
Каждый объектный файл через таблицу релокации понимает какие внешние зависимости необходимы данному объектому файлу, зная эту информацию он находит в других объектный файлах реализацию необходимого ему вызова
Пример:
```
undefined reference to 'foo'
```
Ошибка подобного формата как раз указывает на то, что линковщик **не смог** найти определение функции `foo`. Такая ошибка свидетельствует, что мы либо забыли описать реализацию в нашем объектном файле, либо не подключили нужную библиотеку линковщику.
4) Объединение секций
Можно сказать, что линковщик выполняет `merge` секций объектных файлов.
Пример:
```
multiple definition of 'bar'
```
Ошибка подобного формата явно свидельствует о том, что при объединении секций линковщик нашел 2 реализации одного и того же. 
5) Заполнение таблицы релокаций
Зная карту из пункта 3, линковщик без труда заменяет вызовы функций на необходимые вызовы в других объектныъ файлах, тем самым закрывая "дырки"
6) Подготовка финального исполняемого файла
Теперь, когда у линковщика есть все определения, весь код и все вызову, ему не составит труда сконструировать исполняемый файл. По сути исполняемый файл это "огромный" объектный файл (Это опредление очень утрировано, так как есть пониятие статический и динамической библиотеки)
