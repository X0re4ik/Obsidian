


>`g++ -S driver.ii -o driver.s`


Сам процесс компиляции разбит на несколько этапов:
1) Лекический и синтаксический анализ
2) Синтаксический анализ
3) Семантический анализ
4) Генерация промежуточного представления
5) Оптимизация (на уровне IR)
6) Генерация ассемблерного кода (Кодогенерация)


# Frontend
## Лекический и синтаксический анализ
Плоский поток языка превращается в Абстрактное Синтексическое Дерево (AST - Abstract Syntax Tree)
Пусть имеется строка кода
```C++
int x = 5 + 3;
```
Разбиваем ее на токены
```txt
[INT] [ID("x")] [ASSIGN] [NUM(5)] [PLUS] [NUM(3)] [SEMI]
```
По правилу языка компилятор знает, что это присвоение переменной, итоговый вид будет следующим: 
```
      [VariableDeclaration]
      /         |         \
   [Type]      [ID]    [Expression]
   (int)       ("x")      /    |    \
                       [NUM] [PLUS] [NUM]
                        (5)          (3)
```

## Что с ошибками?
Пусть пользователь допустил ошибку синтаксиса
```C++
int x + 6;
```
Процесс парсинга пройдет успешно, так как *неизвестных* операций для компилятора нет, но на этапе синтаксического анализа будет ошибка и компилятор явно скажет, что я не ожидал такого токена после инициализации
>g++ main.cpp
```bash
main.cpp: In function ‘int main()’:
main.cpp:3:9: error: expected initializer before ‘+’ token
    3 |   int x + 6;
      |         ^
```
Отметим, что синтаксический анализ не проверяет типы, запись формата
```C++
std::string x = 6;
```
Будет с точки зрения лексики и синтаксиса корректна, но вот с точки зрения семантики...
## Пример

```C++
int add(int x, int y) { return x + y; }

int main() {
  int sum = add(5, 6);
  return 0;
}

```

```bash
[TranslationUnit] (Корень файла)
 │
 ├── [FunctionDeclaration] (Объявление функции)
 │    │
 │    ├── [ReturnType] -> [BuiltinType: int]
 │    ├── [Name] -> [Identifier: "add"]
 │    │
 │    ├── [ParameterList]
 │    │    ├── [Parameter]
 │    │    │    ├── [Type] -> [BuiltinType: int]
 │    │    │    └── [Name] -> [Identifier: "x"]
 │    │    └── [Parameter]
 │    │         ├── [Type] -> [BuiltinType: int]
 │    │         └── [Name] -> [Identifier: "y"]
 │    │
 │    └── [BlockStatement] (Тело в фигурных скобках { ... })
 │         └── [ReturnStatement] (return ...)
 │              └── [BinaryExpression] (Операция с двумя ветвями)
 │                   ├── [Operator] -> "+"
 │                   ├── [LeftOperand] -> [Identifier: "x"]
 │                   └── [RightOperand] -> [Identifier: "y"]
 │
 └── [FunctionDeclaration] (Объявление функции)
      │
      ├── [ReturnType] -> [BuiltinType: int]
      ├── [Name] -> [Identifier: "main"]
      ├── [ParameterList] -> (пусто)
      │
      └── [BlockStatement] (Тело в фигурных скобках { ... })
           │
           ├── [DeclarationStatement] (int sum = ...)
           │    ├── [Type] -> [BuiltinType: int]
           │    ├── [Name] -> [Identifier: "sum"]
           │    └── [Initializer] (= ...)
           │         └── [FunctionCallExpression] (Вызов функции)
           │              ├── [Callee] -> [Identifier: "add"]
           │              └── [ArgumentList]
           │                   ├── [Literal] -> [IntLiteral: 5]
           │                   └── [Literal] -> [IntLiteral: 6]
           │
           └── [ReturnStatement] (return ...)
                └── [Literal] -> [IntLiteral: 0]
```

## Семантический анализ
Семантический анализ так же разбит на 2 части:
## Контроль типов
Компилятор проходится по дереву и на каждому узле проверяет возможно ли выполнить приведение типов, если бы в примере выше функция возвращала бы float, то на данном этапе компилятор попытался бы найти правило приведения int -> float и, если такое правило существует, то в исходное дерево добавил бы дополнительную операцию cast-ования
## Разрешение имен
Все это время мы никак не проверяли переменные которые используем, мы только говорили об их объявлении и об их использовании, но видимы ли они компилятору вообще становится ясно только на данном этапе
```C++
int add(int x, int y) { return x + y; }

int main() {
  int sum = add(x, 6);
  int x = 6;
  return 0;
}
```
С точки зрения синтаксиса все логично, но вот семантика нарушена, именно на этапе разрешения имен мы получим ошибку
```bash
main.cpp: In function ‘int main()’:
main.cpp:6:17: error: ‘x’ was not declared in this scope
    6 |   int sum = add(x, 6);
      |                 ^
```
# Middle-end
# Формирование IR кода
На данном этапе компилятор проходится по AST-дереву и выполняет ее преобразование в IR (`Intermediate Representation`) инструкции. Если проводить аналогию, то этап IR - это генерация *универсального* асемблерного кода - код не привязан жестко к какой либо архитектуре, но имеет довольно низкие операции вставки данных в блоки памяти.

>Представление LLVM IR спроектировано для проведения промежуточного анализа и преобразований, которые осуществляются системой оптимизации компилятора. Данное представление было разработано с учетом множества специфических задач, включающих в себя поддержку простых оптимизаций времени исполнения, кроссфункицональных и межпроцедурных оптимизаций, анализа всей программы, агрессивных преобразований реструктурирования, и.т.д. При этом наиболее важным является тот факт, что это представление само является языком с четко заданной семантикой. [https://rus-linux.net/MyLDP/BOOKS/Architecture-Open-Source-Applications/Vol-1/llvm-3.html]
### Пример
>main.cpp
```C++
unsigned add1(unsigned a, unsigned b) {
  return a+b;
}

// Не самый эффективный способ сложения двух чисел.
unsigned add2(unsigned a, unsigned b) {
  if (a == 0) return b;
  return add2(a-1, b+1);
}
```
>main.ll
```text
define i32 @add1(i32 %a, i32 %b) {
entry:
  %tmp1 = add i32 %a, %b
  ret i32 %tmp1
}

define i32 @add2(i32 %a, i32 %b) {
entry:
  %tmp1 = icmp eq i32 %a, 0
  br i1 %tmp1, label %done, label %recurse

recurse:
  %tmp2 = sub i32 %a, 1
  %tmp3 = add i32 %b, 1
  %tmp4 = call i32 @add2(i32 %tmp2, i32 %tmp3)
  ret i32 %tmp4

done:
  ret i32 %b
}
```
### Оптимизация
На данном этапе выполняются команды оптимизации компилятора (`-O1 -O2`). Именно IR этап позволяет компилятору выполнить "ускорение" отдельных частей кода, так например inline функции на этапе IR *могут* (но не обязаны) могут быть проброшены в вызываемый код
>`main.cpp`
```C++
inline int add(int x, int y) {
    return x + y;
}

int foo(int a, int b) {
    int s = add(a, b);
    return s * 2;
}
```
Пример без оптимизации показывает, что RI генерирует вызов функции (в будущем асемблерный код будет схож)
```
%call = call i32 @add(i32 %a, i32 %b)
```
>main.ll (Без оптимизации)
```
define i32 @add(i32 %x, i32 %y) {
entry:
  %sum = add i32 %x, %y
  ret i32 %sum
}

define i32 @foo(i32 %a, i32 %b) {
entry:
  %call = call i32 @add(i32 %a, i32 %b)
  %mul  = mul i32 %call, 2
  ret i32 %mul
}
```
Но аналогичный пример с оптимизацией показывает, что IR избавиться от лишнего вызова `add`, а ее код напрямую будет инстанциорован в код функции `foo`
>main.ll (С оптимизацией)
```
define i32 @foo(i32 %a, i32 %b) {
entry:
  ; бывшее тело @add:
  %sum = add i32 %a, %b

  ; остальной код foo:
  %mul = mul i32 %sum, 2
  ret i32 %mul
}
```
