Перечислим основные виды утечки памяти C++ приложения
1) Явное неосвобождение и потеря указателя
Стандартная ситуация C++ приложения: выделили память через `new` и забыли выполнить `delete`
**Пример:**

```C++
#include <iostream>
#include <random>

int main() {
  std::random_device rd;
  std::mt19937 gen(rd());
  std::uniform_int_distribution<int> dist(1, 100);
  int *x;
  for (int i = 0; i < 1000; i++) {
    x = new int{dist(gen)};

    if (*x > 50) {
      delete x;
    }
  }

  std::cout << x << std::endl;
  return 0;
}
```
2) Утечка из-за исключения
Ранее мы говорили, что ключевая проблема утечка памяти - отсутствие осовобождения ресурсов после их захвата. Идентичная ситуация у нас с выбросом исключения.
```C++
#include <iostream>

int main() {
  std::cout << "Start Program" << std::endl;

  for (int i = 0; i < 1000; i++) {
    try {
      int *x = new int{i};
      if (*x > 500) {
        throw std::runtime_error("x > 500");
      }
      delete x;
    } catch (std::runtime_error) {
      // delete x;
    }
  }
  return 0;
}
```
2) Циклические ссылки
```C++
#include <iostream>
#include <memory>

struct B;

struct A {
  std::shared_ptr<B> _ptrA;
  ~A() { std::cout << "~A()\n"; }
};

struct B {
  std::shared_ptr<A> _ptrB;
  ~B() { std::cout << "~B()\n"; }
};

int main() {
  auto a = std::make_shared<A>();
  auto b = std::make_shared<B>();
  
  a->_ptrA = b; // a ссылается на b
  b->_ptrB = a; // b ссылается на a
  
  return 0;
}
```
4) Отсутствие виртуального деструктора
При использовании полиморфизма **виртуальный деструктор** это обязательная часть приложения. Это отдельная большая тема, но его отсутствие влияет на освобождение ресурсов классом приложения
**Пример**
```C++
#include <iostream>
#include <memory>

class A {
  virtual void print() = 0;
};

class B : public A {
  int *data = new int[100];
  void print() override { std::cout << "B->print()" << std::endl; }

  ~B() { std::cout << "B->~B()" << std::endl; }
};

int main() {
  A *a = new B{};
  delete a;
  return 0;
}
```
Отсутствие виртуального деструктора в классе `~A` приводит к тому, что деструктор класса B просто не вызывается, соответсвенно, утечка памяти.

1) Статический анализ кода.
Статический анализ исполняется на этапе `CI`, когда наше приложение только собирается Преимущества статического анализа - проверка кода без его запуска (проверка происходит на этапе построение AST-дерева). В качестве стандарта де-факто используется `Clang Static Analyzer`, давайте более детально посмотрим на его работу.
**Clang Static Analyzer**
Установка
```bash
sudo apt-get install clang clang-tools
```
Запуск программы для формирования отчёта
```bash
scan-build -o ./reports clang++ main.cpp -o my_app
```

**Вывод**

| Вид утечки                              | Статус |
| :-------------------------------------- | ------ |
| Явное неосвобождение и потеря указателя | ✅      |
| Утечка из-за исключения                 | ✅      |
| Циклические ссылки                      | ❌      |
| Отсутствие виртуального деструктора     | ❌      |
Видим, что статический анализатор поможет вам в разработке, ускорит поиск несложных ошибок, но магии в нем нет. Разобрать тривиальные ошибки неосвобождения памяти - ДА. Разобрать циклические ссылки, выделения памяти в классах - НЕТ.

2) Санитайзеры (Sanitizers)
[Санитайзеры](https://skkv-itmo.gitbook.io/c-cpp-cookies/code-analysis/sanitizer) - инструменты отладки, позволяющие проводить более продвинутые анализы во время исполнения программного обеспечения.
Компилятор Clang умеет выявлять следующие виды санитайзеров:
- [AddressSanitizer](https://clang.llvm.org/docs/AddressSanitizer.html) (`address`) - детектор ошибок при работе с адресацией (выход за границы выделенной области памяти, стека, двойное высвобождение памяти и другое).
- [ThreadSanitizer](https://clang.llvm.org/docs/ThreadSanitizer.html) (`todo`) - детектор ошибок при взаимодействиями между потоками (data race)
- [UndefinedBehaviorSanitizer](https://clang.llvm.org/docs/UndefinedBehaviorSanitizer.html) (`todo`) - детектор специфических ошибок, чаще всего регламентированных в стандарте как [_Undefined Behavior_](https://en.cppreference.com/w/c/language/behavior)
- [MemorySanitizer](https://clang.llvm.org/docs/MemorySanitizer.html) (`todo`) - детектор чтения из неинициализированной области памяти.
- [LeakSanitizer](https://clang.llvm.org/docs/LeakSanitizer.html) (`todo`) - детектор утечки (не освобождаемой) памяти
- [DataFlowSanitizer](https://clang.llvm.org/docs/DataFlowSanitizer.html) (`todo`) - обобщенный анализ потока данных
Для работы санитайзера необходимо выставить флаги:
* `-g` - флаг позволяет видеть простой и понятный вывод ошибок (с указанием строк кода)
* `sanitize` - список инструментов отладки через запятую. Отметим, что все виды санитайзеров совместить нельзя, можно только несколько. Невозможно объееденить в одной программе: `address`, `thread`, `memory`
* `no-omit-frame-pointer` - флаг, который позволяет выводить "красивую" картинку по `stack`-у
**Пример**
```bash
clang++ -O1 -g -fsanitize=address -fno-omit-frame-pointer main.cpp -o main
```

| Утечка из-за исключения                 | ✅      |
| Циклические ссылки                      | ✅      |
| Отсутствие виртуального деструктора     | ✅      |
